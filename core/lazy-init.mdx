## Eager init (default)

Default тохиолдолд **бүх singleton bean** контейнер ачаалмагц шууд үүсэж, dependency инжект
хийгдэж, `@PostConstruct` дуудагдана. Энэ нь:

- Стартап үед бүх алдаа илэрнэ ("fail fast")
- Анхны хүсэлтийн latency хамгийн бага
- Нөөц илүү зарцуулах

## Lazy init

`lazyInit=true` үед singleton bean-уудыг **анх удаа `getBean(...)`** дуудагдсан үед л
үүсгэнэ. Урт удаан startup, эсвэл бүх bean-ийг ашиглахгүй сценаринд тохиромжтой.

<CodeGroup>
```java Annotation
@EnableIoC(basePackage = "com.example.app", lazyInit = true)
public class App {}
```

```java Builder
ApplicationContext ctx = ApplicationContextBuilder.builder()
        .basePackage("com.example.app")
        .lazyInit(true)
        .build();
```
</CodeGroup>

```text
[Context] lazyInit=true → singleton bean-уудыг шаардлагатай үед үүсгэнэ.
```

`PROTOTYPE` bean нь түрээсхэн тэр чигээрээ lazy байсан тул `lazyInit` тэдгээрт нөлөөлөхгүй.

## Бенчмарк дээрх ялгаа

`mn.edu.num.app.scope` (4 bean)-ийн жишээн дээрх Devin VM-ийн хэмжилт:

| Сценарь | Mean (µs) | P50 (µs) | P95 (µs) |
| --- | ---: | ---: | ---: |
| `startup-eager` | 1681.27 | 876.12 | 7484.38 |
| `startup-lazy` | 1355.09 | 553.14 | 6108.37 |

<Note>
  Дээрх тоо нь жижиг апп дээрх жишээ. Бодит апп дээр bean тоо болон lifecycle hook-уудаас
  шалтгаалан зөрүү ихсэх боломжтой. Өөрийн апп дээрээ
  `BenchmarkRunner --package <pkg>` командаар турших боломжтой.
</Note>

## Дизайн дүгнэлт

`lazyInit=true` болсон үед циклик dependency илрүүлэх явц хойшилно (анхны бодит lookup
хүртэл). Production-д ашиглах бол startup-ийн нэг хэсгийг lazy болгож, dev/CI horimd
eager хэвээр үлдээх hybrid стратеги тохиромжтой.

```java
boolean lazy = "production".equals(System.getenv("ENV"));

ApplicationContext ctx = ApplicationContextBuilder.builder()
        .basePackage("com.example.app")
        .lazyInit(lazy)
        .build();
```
